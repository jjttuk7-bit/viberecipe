### GAD-1 (가장 큰 결정): 수정 흐름의 의미

| 후보 | 흐름 | 헌법 정합 |
|------|------|----------|
| A | 사용자 수정 = 클라가 즉시 RecipeState mutate. user 메시지 생성 안 함. LLM은 다음 fetch에서 current_state로만 인지 | §1.3 합의 정신 약함 — LLM이 "사용자가 뭘 했는지" 모르고 *결과 상태*만 봄. D-001은 위반 아님 |
| **B+** | 사용자 수정 = (i) 클라가 즉시 시각 mutate (UI 즉각 반응) + (ii) "재료에서 X 뺐어" 같은 user 메시지 자동 생성 → 다음 fetch에서 LLM이 처리 → 정식 new_state 반환 | §1.3 합의 정신 자연 + TASTE §4 친구 톤(LLM이 "그럼 시간도 줄여보자" 같은 합의 반응 가능) + D-001 위반 0 |
| C | B와 동일하되 (i) 클라 즉시 mutate 없음. user 메시지만 자동 추가. LLM 응답 대기 동안 UI 그대로 | §1.3 정합. 단 UI 반응 느림 (LLM 응답 1~5초 대기) |

**architect 권고**: **B+**. 즉각 시각 반응 + 합의 정신 동시 만족. 단점은 *LLM 응답이 사용자 mutate와 다를 가능성* — 그러나 다음 fetch에서 LLM이 splitDiff로 *재조정* 가능 (D-001/D-002 정상 작동). 메시지 표현 예시:
- 재료 삭제: `"재료에서 'X' 뺐어"`
- 재료 양 변경: `"X 양을 'A'에서 'B'로 바꿨어"`
- 게이지 변경: `"매운맛을 5에서 7로 올렸어"`
- 도구 삭제: `"'X' 도구는 빼고 가자"`

## 신규 ADR 후보 (AD.T4 등재)

### D-022. 사용자 직접 수정 — RecipeState의 두 번째 출처

**맥락**: D-001은 "LLM이 new_state만 반환, 코드가 diff 계산"을 명시. D-021은 BUILD UX = 채팅+stage+임베드+옵션. 두 ADR 모두 *사용자가 임베드를 직접 수정*하는 경우를 미명시. 페어 프로그래밍의 핵심인 *사용자 주도*를 BUILD에 박으려면 명시화 필요.

**결정** (GAD-1=B+ 채택 시):
1. RecipeState의 *출처*는 (a) LLM의 `engineResponse.new_state` 또는 (b) 사용자 직접 수정. 두 경우 모두 코드가 diff 계산 (D-001 위반 0).
2. 사용자 수정 흐름:
   - (i) 클라가 즉시 RecipeState 시각 mutate — UI 즉각 반응
   - (ii) "재료에서 X 뺐어" 같은 *user 메시지 자동 생성* 후 messages 배열 추가 (다음 turn에 자동 전송 안 함 — 사용자가 직접 전송하거나 추가 입력)
   - (iii) 사용자가 다음 turn 전송 시 LLM이 변경을 공식 처리 — *재조정된* new_state 반환 가능 (예: 재료 줄면 시간도 줄임)
3. 사용자 수정의 *modified diff 표시는 안 함* (GAD-6) — 자기 행위 reminder 노이즈 회피.
4. *편집 가능 필드 범위* (본 사이클): ingredients / tools / taste / texture 4개 (GAD-2=A).
5. *stage 진행*: 사용자 수정은 stage 변경 트리거 아님 (GAD-7).
6. *모든 직접 수정 행위는 user 메시지로 흔적 남김* — 즉 messages 히스토리만 보면 *누가 무엇을 바꿨는지* 재구성 가능. D-008 용접의 정신과 정합 (모든 행위가 다음 BUILD 입력으로 흘러갈 수 있음).

**이유**:
- §1.3 합의 정신 정합 — 사용자 수정도 *한 turn*으로 본다 (페어 프로그래밍의 자연 흐름)
- D-001 위반 0 — diff 계산은 여전히 코드 (splitDiff). LLM은 여전히 new_state 반환.
- D-002 "수정=diff" 일반화 — 사용자 수정은 *자기 행위*라 표시 노이즈. LLM 처리 후 modified는 표시.
- D-021 4-요소 패턴 확장 — 임베드의 의미가 *읽기 전용 → 인터랙티브*로 확장. D-021 결과 섹션 보강.

### D-023. 에디트 히스토리 — 클라 messages 누적 + RecipeState 스냅샷 1개

**맥락**: 백엔드는 ENGINE.md §3에 따라 최근 8턴만 받는다. 클라 UI도 이를 따라 8턴만 보이면 *이전 합의*를 사용자가 확인 못 한다. 페어 프로그래밍의 git 메타포가 약해진다.

**결정**:
1. *클라 messages 무한 누적* — 세션 안에서 모든 메시지 보존.
2. *백엔드 호출은 여전히 messages.slice(-8)* — ENGINE.md §3 / RequestBodySchema.max(8) 정합.
3. UI 표시:
   - 초기 진입: 최근 4 assistant turns + 사이 user 메시지만 표시 (R-AD-1 화면 과다 누적 방지)
   - "더 이전 보기" 펼침 버튼 상단 — 누른 후 전체 messages 표시
4. *직전 되돌리기*:
   - RecipeState 스냅샷 1개 (`prevSnapshot: RecipeState | null`) + lastDiff 스냅샷 1개 (`prevDiff: SplitDiff | null`) 클라 메모리 보존
   - "직전 응답 취소" 버튼: 입력 행 근처 작게. 누르면 RecipeState/lastDiff/messages를 직전 상태로 되돌림
   - 1단계만 (`prevSnapshot`이 set되면 그 이전은 잃음)
5. *영속 X*: 세션 새로고침 시 초기화. 영속은 ROADMAP P2 `recipe_versions` 사이클.

**이유**:
- "한 단계 뒤로" 가능성이 있어야 사용자가 *자신감 있는 실험* 가능 (페어 프로그래밍의 핵심)
- 영속 보류는 P2 인증 사이클 동기화 — 인증 없이 영속은 D-007 정신 약화

## 잔존 위험 / 가드 (AD.T3 inspector 인계)

- **R-AD-1**: 클라 messages 무한 누적의 메모리 한계 — 일반 BUILD 세션은 수십 턴 이내, 1MB 미만이라 무시 가능. 단 *세션 새로고침 시 손실*은 명시.
- **R-AD-2**: GAD-1=B+ 채택 시 *자동 user 메시지가 LLM 헷갈리게 할 위험* — 권고 표현 형식("재료에서 X 뺐어")이 일반 user 입력과 *구분 가능한지*. ENGINE.md §6 컨벤션 또는 시스템 프롬프트에 명시 필요? 본 사이클에서는 *일반 user 메시지로 처리* (구분 안 함). 다음 사이클에서 시그널 강화 검토.
- **R-AD-3**: GAD-1=B+의 *클라 즉시 mutate*와 *LLM 응답 재조정* 사이 race condition — 사용자가 빠르게 여러 수정 → LLM 응답 도착 시 *덮어쓰기* 위험. 본 사이클: LLM 응답이 클라 mutate를 *덮어씀* (LLM이 SSOT). 명시.
- **R-AD-4**: 직전 응답 취소 버튼이 *messages도 되돌리는가*? 현재 권고: messages도 되돌림 (1단계). 즉 *직전 LLM 응답 메시지*도 사라짐. 사용자 메시지는 보존 (= 입력 텍스트 box로 복원).
- **R-AD-5**: D-022 채택 시 *비편집 필드*(name/concept/steps/time_min)는 본 사이클 *읽기 전용 그대로*. UI 시그널로 *편집 가능/불가능* 명시 (예: 칩에 hover ↔ 텍스트 hover).
- **R-AD-6**: GAD-7 (stage 진행 안 함)이 의도된 동작이지만 *사용자가 mutate 후 LLM 응답을 안 받고* 다음 단계로 가려 하면 어떻게? — 현재 사용자는 "stage를 직접 변경할 UI"가 없음. LLM 응답만이 stage 변경. 자연 정합.
