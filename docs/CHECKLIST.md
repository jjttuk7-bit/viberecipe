## 0. 출발점 — 입력은 예측 불가 (vibe-X의 본질)

사용자는 *무엇이든* 자연어로 던진다:
1. 재료 ("삼겹살 200g") · 2. 음식명 ("김치찌개") · 3. 자세한 요청 ("덜 맵게 5분 안에 술안주") ·
4. 모든 요리권 (한식·양식·일식·…) · 5. 냉털 ("냉장고에 두부·계란·김치밖에") · 6. 모호 ("뭔가 따뜻한 거")

→ 이걸 받아내는 **인테이크 라우터**가 제품의 입구이자 첫 DNA.

### 핵심 설계 원칙 — 구조화 vs LLM (사용자 통찰)
> **신뢰·안전·검증·재현이 필요한 건 구조화. 개방·창의·언어·취향이 필요한 건 LLM. 정답이 중요한 곳은 검증된 데이터로 LLM을 묶는다(grounding).**

| 영역 | 구조화(결정적) | LLM(개방적) |
|---|---|---|
| 입력 이해 | intent/slot **스키마**(JSON) | 임의 자연어 → 슬롯 채우기, 모든 언어 |
| 요리권 | 팩 *인터페이스*(L2) | 임의 요리권 일반화(양식·일식…) |
| 비율/공식 | **검증된 데이터**(해자) | 미수록은 논리 추론(단정 금지) |
| 빌드 흐름 | stage 머신(concept→…→done) | 대화·제안·되묻기 |
| 안전 | 알러지/식품안전 **규칙(하드)** | 설명·대안 제시 |
| 단위/계량 | 변환 **테이블** | 자연어 분량 해석 |
| 맛 조절 | 재료-맛역할 **맵** | 미묘한 취향 판단 |
| 학습 | fingerprint **스키마** | 패턴 해석 |

## 1. 🧬 DNA — 남이 쉽게 못 따라오는 것 (최우선)

- 🟡 **용접 루프**: 빌드 → 요리(런타임) → 패치 → 학습. (남은 레시피만 줌 / 우린 루프를 돌림)
- 🟡 **검증된 포뮬러 지식**: 유케이의 도메인 권위 + 검증 파이프라인. (LLM 일반지식 ≠ 검증된 공식)
- ⬜ **시간이 쌓는 개인 지문**: 쓸수록 깊어지는 부엌 프로필 → 전환비용. (오래 쓸수록 못 떠남)
- 🟡 **구조화×LLM 아키텍처**: 위 원칙을 코드로. (대충 LLM에 다 던지는 클론과 격차)

## 2. 입력 이해 / 라우팅  (지금 가장 큰 공백)

- ⬜ **인테이크 라우터**: 첫 입력 → 구조화 추출 `{intent, ingredients[], dish?, cuisine?, constraints{time,diet,tools,mood,servings}, mode}` (LLM+JSON 스키마)
- ⬜ intent 분기: 재료 / 음식명 / 요청 / 냉털 / 모호 → 각기 다른 진입 흐름
- ⬜ **냉털 모드**: 가진 재료 → 만들 수 있는 것 역제안
- ⬜ 모호 입력 되묻기(1회) vs 합리적 가정 후 진행
- ⬜ **다국어 입력** 수용(영어·기타) — 출력 언어 일치
- ⬜ 멀티모달: 냉장고/완성 **사진**, **음성** 입력
- ⬜ "모르는/위험한 요청" 우아한 처리

## 3. 빌드 루프 (대화형)

- ✅ concept→base→taste→steps→done stage 머신
- ✅ JSON 모드 구조화 출력 + 스키마 검증(D-001/D-004)
- ✅ 실시간 스트리밍(저지연) + 즉시 전환
- ✅ 기본-먼저 추천 / 복수·단일 선택(options_mode)
- ✅ 완성 마무리 + 레시피 복사
- ⬜ **임의 요리권 일반화** (지금 코어는 일반이나 팩은 K-food뿐 → 양식/일식 팩 또는 무팩 일반 모드 검증)
- ⬜ 사용자 직접 수정(재료/게이지) 흐름 재점검(데모에서 일부 빠짐)

## 4. 지식 / 검증 (해자 엔진)

> ★ **원칙: K-Food Formula 는 *하나의 소스(예시)*일 뿐 절대 기준이 아니다.** 다른 요리권에 강요 금지(프롬프트가 "참고서일 뿐, 한식 아니면 일반화" 로 reframe됨). 최종 지향: **RAG 다출처 retrieve → LLM 최종 검수**. K-Food 는 그중 첫 인덱싱 소스.
- ⬜ **RAG 지식 계층**: 여러 소스(공식·웹·DB)에서 관련 지식 retrieve → 주입. K-Food 팩은 정적 첫 버전.
- ⬜ **LLM 최종 검수**: 생성 레시피를 retrieve된 근거 + 안전 기준에 대조해 LLM 이 검토(self-review).

- ✅ L2 지식 팩 구조(`lib/formulas.ts`, 교체 가능) — 코어 요리권 무관(CLAUDE §1.5)
- 🟡 K-Food 12 공식: 2/12 추출(고추장·김치찌개), `verified:false`
- ✅ 검증 토대: `source`+`verified` 플래그, 검증된 것만 "공식" 단정, 지어내기 금지
- ⬜ **추출+교차검증 스크립트**(이미지→구조화→2차 대조) → 초안
- ⬜ 유케이 확인 도장 → `verified:true` (나머지 10 공식)
- ⬜ 재료-맛역할 맵 **요리권별** 확장(지금 한식만)
- ⬜ **출력 검증**: 생성 레시피가 공식/안전을 지키는지 코드 대조

## 5. 런타임 / 요리 (차별점 — 지금 빠짐)

- ⬜ **쿡 모드**: 스텝 진행 + 타이머 + 핸즈프리(음성 "다음")
- ⬜ **런타임 패치/디버그**: "너무 짜" → 변경 **diff**로 즉석 수정 (지금 대화로만)
- ⬜ **포스트모템**: 30초 회고 → 결과 기록
- ⬜ 핫픽스 → 회고 → 학습으로 흐르는 용접 강제

## 6. 학습 / 개인화 (시간 해자)

- 🟡 부엌 지문(localStorage) — 피드백→trait→다음 빌드 주입
- ⬜ **DB 영속**(로그인) — 기기/세션 넘어 유지 (지금 로컬뿐)
- ⬜ 지문 성장 가시화(자라는 게 보이게)
- ⬜ 집단지성: n명 같은 실패 → 레시피 자가진화(네트워크 효과, P3)

## 7. 안전 / 신뢰 (음식이라 필수)
